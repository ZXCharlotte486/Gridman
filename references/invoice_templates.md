# 发票与票据结构化提取模板

> 定位：MinerU OCR 输出 Markdown 后，AI 按本模板逐字段提取，输出结构化数据
> 来源：来也 ADP 字段规范（32 种国内票据 + 全球发票 + 采购订单）
> 性质：实操

---

## 一、使用流程

```
用户提供发票/票据文件
        ↓
Step 1: document_ocr → 输出 Markdown 文本
Step 2: 读取本模板，确定票据类型
Step 3: 按对应字段清单逐项提取
Step 4: 输出结构化结果（JSON 或 Excel）
        ↓
输出到 gridman-mind/outputs/
```

**关键规则：**
- 先判断票据类型，再按对应模板提取（不同票据字段不同）
- 未识别到的字段标记为空，不编造
- 金额类字段保留原始数字格式（含小数点）
- 日期统一为 YYYY-MM-DD 格式
- 如果 OCR 文本模糊导致无法确定某字段，标注"[不确定]"

## 三、国内增值税发票字段模板

适用于：增值税专用发票、增值税普通发票、电子发票（专票/普票）

### 必提字段

| 字段名 | 说明 | 提取要点 |
|--------|------|---------|
| 发票代码 | 10位或12位数字 | 通常在发票右上角 |
| 发票号码 | 8位数字 | 紧邻发票代码 |
| 开票日期 | YYYY年MM月DD日 | 转为 YYYY-MM-DD |
| 购买方名称 | 企业全称 | "购买方"或"购方"标签后 |
| 购买方纳税人识别号 | 15-20位 | "纳税人识别号"或"统一社会信用代码" |
| 销售方名称 | 企业全称 | "销售方"或"销方"标签后 |
| 销售方纳税人识别号 | 15-20位 | 同上 |
| 金额合计 | 不含税金额 | "金额"列合计，注意区分"价税合计" |
| 税额合计 | 税额 | "税额"列合计 |

| 价税合计（小写） | ¥数字 | 同行小写金额 |

## 八、常见问题处理

| 情况 | 处理方式 |
|------|---------|
| OCR 文本中金额有多个，分不清哪个是"价税合计" | 找"价税合计"或"（大写）"标签，小写金额通常带 ¥ 符号 |
| 购买方和销售方搞反 | 购买方在上/左，销售方在下/右（国内发票标准版式） |
| 明细行错位 | 按列对齐逻辑匹配，数量×单价=金额 可做校验 |
| 电子发票没有发票代码 | 全电发票（新版）确实没有发票代码，只有发票号码（20位） |
| 多张发票在一个 PDF 里 | 按页拆分，每页独立提取 |
| OCR 识别错误（如 0 和 O 混淆） | 金额字段只保留数字和小数点；税号只保留数字和大写字母 |

## 九、发票批量整理流程（报销/抽凭场景）

> 本章是"成批发票从识别到归集"的端到端流程，区别于前面"单张发票字段提取"。
> 古立特用 `document_ocr`（识别）+ `pdf_merge`（合并）落地，不依赖任何第三方工具。

### 9.1 发票合并归集流程

> 类型：流程
> 来源：实务 SOP 提炼
> 置信度：实务经验

[场景：报销/装订前，把一堆零散发票（PDF + 图片）去重、排序、合并成一个可打印 PDF]

四步走，逐步确认，不跳步：

**Step 1 检索发票文件**
扫描目标目录，范围含 PDF 和常见图片格式（jpg/png/jpeg）。通过文件名关键词（发票、invoice、fapiao）+ 内容特征初筛出候选发票，列清单让用户确认范围。

**Step 2 识别发票信息**
对候选文件逐一 OCR/文本解析（调 `document_ocr`），按本文件第三章字段模板提取四个关键字段：发票号码、开票日期、价税合计金额、销售方名称。

⚠️ 识别失败的文件**单独列出并标红，不静默丢弃**——漏一张发票在报销/抽凭里就是缺凭证。

**Step 3 去重与排序**

去重分两层处理，**不静默删除任何疑似发票**：

- **第一层·文件完全相同（可自动清理）**：两个文件内容哈希一致 = 同一文件的副本，自动保留一份即可（这是清理副本，不涉及发票判断）。
- **第二层·发票号/金额相同但文件不同（标红，人工确认）**：发票号码相同、或金额+开票日期+销售方都相同，但文件本身不同（如 PDF 原件 vs 手机拍照、或 OCR 可能误判）→ **标红列为"疑似重复组"，列出每组对应的文件、号码、金额，让用户确认**，不自动删。

⚠️ 为什么不自动去重：发票号相同不一定是真重复——可能是原件与扫描件并存、也可能是 OCR 把号码读错把两张不同的票误判成同一张。一旦自动删错，报销就少报一笔、抽凭就漏一张凭证，且用户不知情。"删不删"是有后果的判断，交给人。机器只负责筛出可疑组（机器筛查、人工研判）。

- 排序：按**开票日期**从早到晚排序。
- 输出整理后清单（编号/日期/号码/金额/文件名/状态），状态列标明：正常 / 疑似重复（同组标同一编号）/ 识别失败。让用户核对后再进入合并。

**Step 4 合并输出 PDF**
- 按上一步顺序，图片格式先转 PDF 页，再合并成 1 个 PDF（调 `pdf_merge`）。
- 版面：每张 A4 按需布局（如竖向 1×2 两张发票/页，便于打印装订）。
- 命名：`发票汇总_YYYYQX.pdf`。
- 附首页目录（编号-日期-金额-合计）。

**完成后汇报**（统计概览）：给出一组数字让用户报销/装订前心里有数——
- 候选文件总数 / 成功识别张数 / 识别失败数
- 疑似重复组数 + 涉及张数 + 重复金额
- 去重后实际张数 + 去重后合计金额
- 识别失败/异常需手动处理的文件清单

⚠️ 打印版面（边距、每页张数）必须在指令里固定下来——否则每次重新生成的合并脚本布局会漂移（有边距/无边距不稳定）。

### 9.2 发票分类归档流程

> 类型：流程
> 来源：实务 SOP 提炼
> 置信度：实务经验

[场景：按发票类型分目录归档，便于进项认证/账务处理]

**Step 1 检索**：扫描目标目录（PDF + jpg/png/jpeg）。

**Step 2 识别类型**：逐一 OCR，按第二章"票据类型判断规则"识别类型——增值税专用发票（专票）/ 增值税普通发票（普票）/ 电子发票 / 交通票据等。

**Step 3 分目录归档**：按类型把发票归到不同目录。

⚠️ 航空行程单、铁路客票等**交通票据应单独归类**，不混入专票/普票目录——它们的进项处理规则不同（如旅客运输进项按行程单计算抵扣）。
